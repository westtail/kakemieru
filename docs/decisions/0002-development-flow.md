# ADR-0002: 開発フロー運用方針

- 日付: 2026-03-21
- ステータス: 決定済み

---

## コンテキスト

Issue作成からデプロイまでの一連の開発フローを定義する必要があった。
個人開発だが、業務水準の規律を保ちながら、無駄なオーバーヘッドを避けたい。

---

## 決定事項

### 1. Issue駆動開発

**決定**: すべての作業はIssueから始める

**理由**:
- 作業の目的と完了条件を明確にする
- ブランチ名にIssue番号を含めるため、先にIssueが必要
- 工数記録の単位をIssueに統一する

### 2. ADRの作成タイミング

**決定**: 以下の場合にのみADRを作成する

- 技術的な方針を決定するとき
- 複数の選択肢を検討したとき
- 後で「なぜこうしたか」を残したいとき

**理由**:
- すべての決定にADRを書くと管理コストが高い
- 自明な決定にはADRは不要

### 3. TDD（テスト駆動開発）

**決定**: RED → GREEN → REFACTOR のサイクルで実装する

**理由**:
- テストを先に書くことで仕様を明確化できる
- リファクタリング時の安全網になる
- カバレッジ目標80%を達成しやすい

### 4. コミット粒度

**決定**: 小さい単位でこまめにコミット。wipコミットを許容する

**理由**:
- 問題発生時の切り分けが容易
- revertをコミット単位で細かくできる
- commitlintは不採用（ADR-0001参照）

### 5. リリース単位

**決定**: 機能単位でdevelop → main へPRを作成し、タグを切る

**理由**:
- 問題発生時にどの機能が原因か特定しやすい
- revertが機能単位でできる
- SemVerでバージョン管理することでリリース履歴が残る
